# Инструкция по использованию V4 Переводчика

Переводчик художественных текстов на основе LLM. Работает в два этапа: подготовка контекста (глоссарий) и умный перевод с циклом проверки качества.

## Предварительные требования

1.  Установленный **Node.js**.
2.  Установленные зависимости:
    ```bash
    npm install
    ```
3.  Запущенные локальные сервера с LLM (если используется локальный режим) или API ключи для облачных провайдеров.

---

## Конфигурация

Настройки моделей находятся в файле `src_v4/config.js`.
*   **logic_model**: Основная модель для перевода и проверки (локальный LLM сервер).
*   **fast_model**: Модель для быстрых задач.
*   **google_model**: Настройки для Google Gemini.
*   **groq_model**: Настройки для Groq API.

Для облачных провайдеров экспортируйте ключи:
```bash
export GOOGLE_API_KEY="ваш_ключ"
export GROQ_API_KEY="ваш_ключ"
```

## Рабочий процесс

### Этап 1: Подготовка контекста

Программа читает книгу, разбивает на чанки, извлекает имена и термины, затем создает единый глоссарий.

**Команда:**
```bash
node src_v4/main.js --stage=1 --file=txt/My_Book.txt
```

С указанием провайдера:
```bash
node src_v4/main.js --stage=1 --file=txt/My_Book.txt --model=google
```

**Результат:**
*   `My_Book_project_state.json` — прогресс и состояние.
*   `My_Book_glossary.json` — глоссарий терминов.

**Устойчивость к ошибкам:**
*   Если LLM не вернула ответ (нехватка контекстного окна, таймаут) — извлечение повторяется до 3 раз.
*   Неудачные чанки не помечаются как обработанные — при повторном запуске они будут обработаны снова.
*   Консолидация дополняет существующий глоссарий, а не перезаписывает его.

### ⚠️ Ручная правка глоссария

После завершения Этапа 1 откройте файл `My_Book_glossary.json`:
*   Удалите мусор и обычные слова.
*   Исправьте неверные переводы имён и терминов.
*   Сохраните файл — он будет использоваться как «истина» при переводе.

---

### Этап 2: Перевод

Каждый чанк проходит через цикл: **перевод → проверка → решение → исправление/переперевод**.

**Команда:**
```bash
node src_v4/main.js --stage=2 --file=txt/My_Book.txt
```

С облачным провайдером:
```bash
node src_v4/main.js --stage=2 --file=txt/My_Book.txt --model=google
node src_v4/main.js --stage=2 --file=txt/My_Book.txt --model=groq
```

**Логика цикла проверки:**

| Результат проверки | Действие |
|---|---|
| Оценка ≥ 9.1 и `like = 1` | ✅ **Принят** — сохраняем |
| Оценка ≥ 7.5 и `like = 1` | 🔧 **Доработка** — исправляем ошибки |
| Оценка < 7.5 или `like = 0` | 🔄 **Заново** — перевод с нуля |

*   Максимум 10 итераций на чанк.
*   При исчерпании попыток сохраняется лучший вариант из истории.
*   Прерванный процесс продолжит с того же места при повторном запуске.

**Результат:**
*   По завершении автоматически создается `txt/My_Book_rus.txt`.

## GUI

Веб-интерфейс поверх того же конвейера (CLI продолжает работать как раньше):
```bash
npm run gui
```
Откройте `http://127.0.0.1:3457`. Возможности:
*   **Дашборд** — все проекты с прогрессом, новые книги из `txt/`.
*   **Глоссарий** — табличный редактор вместо ручной правки JSON: поиск, типы, род, заметки, счётчик вхождений термина в чанках (0 — кандидат на удаление).
*   **Монитор** — запуск/остановка этапов, живой лог, карта чанков с цветовой кодировкой статусов.
*   **Чанк** — оригинал и перевод рядом, ручное редактирование, «принять» или «сбросить», полная история попыток с оценками.

Порт настраивается переменной `GUI_PORT`. Сервер слушает только `127.0.0.1`.

## Дополнительные инструменты

### Принудительный экспорт
Собрать книгу из того, что уже переведено (даже если перевод не завершён):
```bash
node src_v4/main.js --stage=export --file=txt/My_Book.txt
```

### Сброс перевода
Начать Этап 2 заново, сохранив результаты Этапа 1 (извлечённые термины):
```bash
node src_v4/tools/reset_to_stage1.js --file=txt/My_Book.txt
```
*Создаётся бэкап текущего состояния перед сбросом.*

## Параметр `--file`

Параметр `--file` **обязателен** на всех этапах. Он определяет:
1.  Какой файл переводить (на Этапе 1).
2.  Префикс для `project_state.json` и `glossary.json`.
3.  Имя выходного файла перевода.

Это позволяет вести **несколько проектов параллельно**:
```bash
# Проект 1
node src_v4/main.js --stage=2 --file=txt/Book_One.txt --model=google

# Проект 2 (независимо)
node src_v4/main.js --stage=2 --file=txt/Book_Two.txt --model=local
```

Каждый проект имеет свои `Book_One_project_state.json`, `Book_One_glossary.json` и т.д.
